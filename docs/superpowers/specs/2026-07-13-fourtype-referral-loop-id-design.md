# FourType Referral Loop and Bahasa Indonesia Design

Date: 2026-07-13

## Objective

Increase completed FourType quizzes generated by existing users inviting friends. The experience must remain free, require no account, and give both people a useful comparison after the friend finishes.

The same core journey will launch in Bahasa Indonesia so Indonesian-speaking visitors can take, share, and compare results without falling back to English.

## Success Measures

The primary metric is referred quiz completions divided by completed quizzes. Supporting metrics are:

- result-to-invite share rate;
- invite open rate;
- invite-open-to-quiz-start rate;
- referred quiz completion rate;
- comparison view rate after a referred completion;
- pair-card share rate;
- completion rate by locale and chapter;
- invalid or expired share-link rate.

Tracking failures must never interrupt the quiz, result, sharing, or comparison experience.

## Scope

### Included

- Full quiz funnel instrumentation from start through comparison sharing.
- One dominant result-page action: challenge a friend and compare.
- A referred quiz entry state that shows the inviter's character and explains the reward for finishing.
- A completed side-by-side comparison for both FourType results.
- A shareable pair result card and pair-specific share copy.
- Referral attribution carried through the quiz without accounts or cookies being required.
- Bahasa Indonesia homepage, quiz questions, result copy, share invitation, comparison experience, dynamic share metadata, and core SEO routes.
- Graceful handling of invalid share IDs and analytics failures.

### Excluded

- Email capture, email delivery, or nurture sequences.
- Accounts, passwords, leaderboards, points, prizes, or public profiles.
- Translating the entire English blog archive into Bahasa Indonesia.
- Claims that a comparison predicts relationship success or compatibility.
- Storing names, answers, or scores in a new database.

The existing email form will be removed from the result journey so it does not compete with the friend invitation. Existing Google Sheet lead records and server code can remain untouched unless they become dead code after the UI removal.

## Recommended Architecture

Use the existing encoded share ID as the referral identity. It already contains the inviter's display name, blend key, and score spread. The referral URL will continue to use the existing comparison parameter:

`/quiz?compare=<shareId>`

No referral database is required. The quiz reads and validates the share ID, keeps the decoded inviter result in client state, and displays the comparison after the friend completes. Analytics records the same opaque share ID so funnel events can be joined without storing quiz answers in a separate system.

This architecture keeps the loop fast and private. Old or malformed IDs fall back to the ordinary quiz with a small neutral notice rather than an error page.

## User Journey

### 1. Original result

The result reveal keeps the accurate identity content, but the first action panel becomes a focused referral prompt:

- headline: challenge a friend who knows you well;
- value: finish both tests to reveal how the two types connect and clash;
- primary action: native share when available;
- secondary action: copy comparison link;
- optional actions such as the story card remain lower on the page.

The existing email form is removed from this page.

### 2. Referred entry

When a valid `compare` parameter is present, the quiz introduction shows:

- the inviter's character image and result name;
- the inviter's display name after safe escaping and truncation;
- a concise promise: complete the test to reveal the pair dynamic;
- one primary button to begin the normal quiz.

The 40 questions and scoring model do not change. Referral context remains visible only where it helps motivation and must not bias answer choices.

### 3. Pair comparison

After the friend completes the quiz, the result page adds a dedicated comparison section above secondary reading links. It presents:

- both character portraits, names, blend labels, and score summaries;
- strongest shared quality;
- natural complement;
- likely friction under pressure;
- communication advice for each person;
- one practical challenge they can try together;
- a reminder that the content is reflective guidance, not a prediction or diagnosis.

Comparison copy is deterministic from the two blend records. Pair order must not create contradictory advice: A compared with B and B compared with A use the same shared interpretation while person-specific communication tips retain their correct direction.

### 4. Pair sharing

The comparison section provides a vertical pair card and share action. The card includes both characters, both result names, one memorable pair insight, and the FourType URL. Share copy invites another friend to take the test rather than claiming a scientifically proven match.

## Analytics

Extend the existing non-blocking `/api/events` pipeline and Google Sheets Events tab. Events use a controlled allowlist and common fields.

### Events

- `quiz-start`: a visitor begins the quiz;
- `chapter-complete`: questions 12, 19, and 24 are completed;
- `quiz-complete`: all 40 answers are completed before result rendering;
- `quiz-result`: a result is displayed, preserving the existing historical event name;
- `invite-share`: native friend invitation is completed;
- `invite-copy`: comparison link is copied;
- `invite-open`: a valid comparison link is opened;
- `referred-quiz-start`: a referred visitor begins;
- `referred-quiz-complete`: a referred visitor finishes;
- `compare-result`: both results are displayed together, preserving the existing historical event name;
- `pair-share`: the pair result is shared;
- `pair-copy`: the pair link or copy is copied;
- `invalid-share-id`: a comparison parameter cannot be decoded.

### Common fields

- locale;
- current blend key when known;
- inviter blend key when known;
- opaque inviter share ID when present;
- event source;
- current route;
- chapter or question number where relevant;
- user agent, captured server-side as it is today.

No email address is attached to referral events. Analytics requests use `sendBeacon` where possible and remain fire-and-forget.

Existing `share-click` and `copy-link` events remain accepted for historical compatibility while new UI uses the more specific invite and pair event names.

## Bahasa Indonesia

Add `id` as a supported locale using the same centralized copy patterns as English, Simplified Chinese, and Spanish.

### Routes

- `/id`
- `/id/quiz`
- `/id/temperament-test`
- `/id/four-temperaments-test`

Localized routes use canonical URLs and reciprocal `hreflang` alternates for `en`, `zh-CN`, `es`, and `id`. The sitemap includes the new routes.

### Translation coverage

- navigation and homepage;
- all 40 questions, instructions, section names, progress messages, and loading copy;
- all result labels and 15 blend summaries;
- friend invitation and referred entry copy;
- pair comparison labels and share copy;
- lead SEO page copy, metadata, FAQ content, and structured data;
- dynamic Open Graph titles and descriptions for shared results.

Character names remain branded as The Commander, The Bard, The Strategist, and The Guardian where brand consistency matters, with clear Indonesian temperament names beside them: Koleris, Sanguinis, Melankolis, and Plegmatis.

Translations should sound natural and direct rather than literal. Safety language must preserve the distinction between self-reflection and clinical assessment.

## Component Boundaries

### Referral context

A small referral-context module owns decoding, validation, and normalized inviter data. Quiz UI consumes the normalized result and does not parse URL data itself.

### Analytics

The analytics helper owns event names and payload shape. Quiz and result components call typed event helpers instead of constructing arbitrary payloads.

### Comparison presentation

Comparison logic remains separate from rendering. The existing comparison module will expose a normalized pair insight object. The comparison UI and pair card consume that object.

### Localization

The locale registry owns supported locale types and route mapping. Homepage, quiz, result, referral, comparison, and share metadata retrieve copy from locale-specific dictionaries without conditional strings scattered through components.

## Error Handling

- Invalid share ID: record `invalid-share-id`, remove comparison context, and allow a normal quiz.
- Unsupported blend in an old link: show the ordinary result and omit comparison.
- Clipboard or native share failure: retain copy-link fallback and visible feedback.
- Analytics endpoint failure: ignore it after a best-effort request.
- Missing localized string: fail during type checking or build verification; do not silently ship partially translated core screens.
- Image generation failure: keep the normal link-sharing action available.

### Automated checks

- Unit coverage for symmetric pair insights and directional communication advice.
- Event payload tests for ordinary and referred quiz paths.
- Locale completeness checks for all supported quiz/result keys.

### Browser checks

- valid referred completion and comparison;
- invalid comparison link fallback;
- native-share fallback behavior;
- pair card layout with the longest result names;
- full Bahasa Indonesia homepage, quiz, result, and comparison journey;
- no layout overlap at common mobile widths;

### Production checks

Probe the new Indonesian routes, sitemap alternates, analytics endpoint allowlist, valid share page, referred quiz URL, and resulting comparison after deployment.

## Rollout

Ship in four internally ordered slices within one implementation:

1. Typed funnel analytics and referral attribution.
2. Referred entry, improved comparison, and pair sharing.
3. Removal of the result-page email distraction and referral CTA refinement.
4. Bahasa Indonesia core funnel and SEO routes.

The feature does not require migration or a new persistence service. Existing share URLs remain valid.
